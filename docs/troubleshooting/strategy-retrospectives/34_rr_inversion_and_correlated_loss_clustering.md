## 1. 문제 상황

### 1.1 트리거
- 1개월 실운영 데이터(38건 거래) 분석 결과, **구조적 손실 패턴** 발견
- 평균 P&L: -0.33%/거래, 누적 -1.87% (-18,700원 on 100만원 seed)

### 1.2 증상

#### R:R(Risk:Reward) 역전
| 지표 | 값 |
|------|-----|
| 평균 수익 (avg win) | +1.28% |
| 평균 손실 (avg loss) | -3.86% |
| R:R 비율 | 1:3.02 (수익의 3배 손실) |
| 손익분기 승률 요구 | 73.5% |
| 실제 승률 | 66.7% |
| 갭 | -6.8%p → 구조적 손실 지속 |

#### 상관 손실 클러스터링
- STOP_LOSS 9건 중 **7건이 2~3심볼 동시 발생**:

| 날짜 | 동시 SL | 심볼 | PnL 합계 |
|------|--------|------|---------|
| 2/27 | 2건 | XRP(-4.33%), SOL(-3.09%) | **-7.42%** |
| 2/28 | 3건 | ETH(-4.01%), XRP(-4.02%), DOGE(-4.29%) | **-12.32%** |
| 3/22 | 2건 | SOL(-4.17%), ETH(-4.03%) | **-8.20%** |

- 3개 클러스터(7건)의 손실 합계: **-27.94%** → 전체 STOP_LOSS 손실의 대부분
- **핵심 발견**: 개별 거래의 exit 로직이 아니라 **동시 노출(exposure)**이 손실 집중의 주 원인

#### Exit 구조 미흡
| Exit Reason | 건수 | avg PnL |
|-------------|------|---------|
| RSI_OVERBOUGHT | 24 | +1.26% |
| STOP_LOSS | 9 | -3.86% |
| TIME_LIMIT | 3 | -2.24% |
| TAKE_PROFIT | 2 | +3.39% |

- TIME_LIMIT 3건은 "MA20 도달 후 하락"으로 조기 익절 가능했던 거래
- BB 중심선(MA20) 도달 시 익절 exit이 없어 수익 → 손실 전환

### 1.3 영향
- **리스크 집중**: 단일 시장 이벤트로 계좌 -12.32% 하락 가능 (2/28)
- **수익성 구조**: R:R 역전으로 승률 66.7%에서도 손실 지속
- **AI 필터 의존**: Rule Engine 1,361건 → AI Confirm 26건(1.9%), 진입 조건 너무 관대

## 2. 원인 분석

### 가설 검증 과정

#### 가설 1: 진입 시 거래량 급증 (volume_surge) ❌ 폐기
- 초기: "STOP_LOSS 진입의 avg vol_ratio = 38.14x → 패닉 구간 진입"
- **반증**: 38.14x는 청산 시점 vol_ratio. 진입 시점 avg = 0.81x (97% < 2x)
- STOP_LOSS vs RSI_OVERBOUGHT 진입 지표 비교: RSI14, RSI7, vol_ratio **모두 동일 분포**
- → 진입 필터로는 구분 불가

#### 가설 2: 개별 거래 SL% 과대 ❌ 효과 미미
- 33번 작업에서 SL 4%→2%, min_profit 상향, rsi_overbought 조정 3가지 시도
- 백테스트 결과 모두 개선 미미 → 개별 파라미터 문제가 아닌 구조적 문제

#### 가설 3: 동시 노출(concurrent exposure) ✅ 핵심 원인
- MAX_CONCURRENT_POSITIONS = 5 (YAML override, Python 기본값 3으로 오인)
- 시장 급락 시 3심볼 동시 SL → 단일 이벤트로 -12.32% 발생
- 동시 포지션 2개로 제한하면 구조적으로 3심볼 동시 SL 차단

#### 가설 4: BB 중심선 도달 시 익절 없음 ✅ 보조 원인
- SIDEWAYS 평균 회귀 전략: BB 하단 → MA20(평균) 회귀가 목표
- MA20 도달해도 exit 없음 → 이후 하락 시 TIME_LIMIT(-2.24%) 또는 SL(-3.86%)로 전환
- **주의**: SL 거래는 대부분 "한 번도 MA20 도달 못함" → BB_MIDLINE_EXIT의 SL 방지 효과는 제한적
- **실제 효과**: TIME_LIMIT 손실 축소 + 수익 확보 빈도 증가

## 3. 해결 방안

### 3.1 Phase 1: 동시 포지션 제한 + 포지션 사이즈 조정

| 항목 | Before | After |
|------|--------|-------|
| MAX_CONCURRENT_POSITIONS | 5 | **2** |
| max_position_size | 20% | **30%** |
| 최대 동시 노출 | 100% (5×20%) | **60%** (2×30%) |
| worst case 동시 SL | -12% (3×4%) | **-8%** (2×4%) |

**설계 근거**:
- 동시 2개로 3심볼 동시 SL 구조적 차단
- 30%로 상향하여 자본 활용률 개선 (40% → 60%)
- 계좌 레벨 worst case: 동시 2건 × 4% × 30% = -2.4% (기존 3건 × 4% × 20% = -2.4%와 동일)

**대안 비교**:
| 대안 | 채택 | 트레이드오프 |
|------|------|------------|
| 하드캡 2개 (채택) | ✅ | 구현 1줄. 상승장 기회 손실 |
| 상관 자산 그룹별 제한 | 보류 | 정교하지만 상관도 계산 복잡 |
| 동일 시간대 진입 쿨다운 | 보류 | 1h 밖 동시 보유는 여전히 가능 |
| MAX=1 | ❌ | 수익 거래도 절반 감소 |

### 3.2 Phase 2: BB_MIDLINE_EXIT + RSI 우선순위 변경

| 항목 | Before | After |

**가드 수치 비교** (백테스트):

|------|--------|---------|-----------|------|

**RSI 우선순위 변경 논리**:

## 5. 교훈 및 배운 점

### 5.1 데이터 분석의 중요성
- 초기 "vol_ratio 38.14x = 진입 시 패닉" 가설이 **청산 시점 데이터 오독**이었음
- 진입 vs 청산 시점을 분리하지 않으면 완전히 잘못된 방향으로 해결책을 설계할 수 있다
- **교훈: 지표를 볼 때 "언제의 값인지" 항상 확인**

### 5.2 YAML override 주의
- Python 코드의 기본값(3)이 아닌 YAML override(5)가 실제 운영값이었음
- 계획 수립 시 "코드 기본값"이 아닌 "실제 런타임 값"을 확인해야 함
- **교훈: 설정 파일이 있는 시스템에서는 반드시 실제 로드되는 값 확인**

### 5.3 백테스트 가드 수치 비교의 가치
- 0.3% 가드가 직관적으로 좋아 보였으나, avg_win +0.63%로 급감 (기존 +2.38%)
- 수치 비교 없이 적용했다면 수익 구조가 심각하게 훼손
- **교훈: 파라미터 변경 시 반드시 여러 값을 비교하고, 2차 효과(avg_win 변화)까지 확인**

### 5.4 포트폴리오 레벨 리스크의 중요성
- 개별 거래 최적화(SL%, TP%)보다 **동시 노출 관리**가 더 큰 영향
- 33번에서 개별 파라미터 3가지 시도 → 모두 미미 → 구조적 문제였기 때문
- **교훈: 개별 거래 최적화 전에 포트폴리오 레벨 리스크부터 점검**

### 5.5 AI 프롬프트에서 금지어 vs 허용어
- "RSI 쓰지 마세요" → LLM이 RSI를 떠올리는 역효과 (ironic process theory)
- "캔들 구조만 허용합니다" → 허용 범위를 좁히는 방식이 더 안정적
- 외부 피드백(Gemini, 데이터 분석가)으로 확인됨
- **교훈: LLM 프롬프트에서는 금지 목록보다 허용 목록이 효과적**
