## 0. 트리거(Why started)
- 운영/설계 논의에서 무엇이 확인됐는지:
  - Next.js 점진 이관은 필요하지만, 이관 초기부터 FE 레포를 분리하면 동기화 비용이 커질 수 있음.
  - 장기적으로는 FE/BE 분리가 바람직하다는 방향성은 합의됨.
- 왜 계획이 필요한지:
  - “분리할지 여부”가 아니라 “언제/어떻게 분리할지”를 명확히 하지 않으면 재작업 위험이 큼.

## 1. 문제 요약
- 증상:
  - 초기 단계 즉시 분리 시 API 계약 변화/배포 동기화로 운영 복잡도가 증가할 가능성
- 영향:
  - 기능: 점진 이관 속도 저하 가능
  - 운영: 두 레포 릴리즈 순서/버전 맞춤 비용 증가
  - 품질: 계약 미안정 상태에서 분리하면 회귀 추적이 어려움

## 2. 목표 / 비목표
### 2.1 목표
1. FE/BE 레포 분리의 **게이트 조건**을 수치 기준으로 정의
2. 분리 전/후 배포 책임 경계를 명확히 정의
3. 23번 이관과 충돌하지 않는 최소 리스크 분리 절차 수립

### 2.2 비목표
1. 본 계획에서 즉시 레포 분리 실행은 수행하지 않음
2. AI 기능 서비스 분해(마이크로서비스화)는 본 계획 범위 밖

## 3. 아키텍처 선택/대안/트레이드오프
- 최종 선택(예정):
  - **모노레포 시작 → Next.js 안정화 후 FE 분리 → 필요 시 BE 분리**

- 고려 대안:
  1) 즉시 FE/BE 동시 분리
  2) FE만 즉시 분리
  3) 모노레포 유지 후 단계적 분리(채택)

- 대안 비교:
  1) 즉시 동시 분리:
    - 장점: 구조 분리가 빠름
    - 단점: 초기 이관과 분리 리스크가 동시에 발생
  2) FE 즉시 분리:
    - 장점: FE 독립성 빠르게 확보
    - 단점: 초기 API 계약 변동기에 동기화 비용 큼
  3) 단계적 분리(채택):
    - 장점: 계약 안정화 후 분리로 재작업 감소
    - 단점: 단기간 모노레포 유지 필요

1. 배포 단위 표준화(이미지 태그/릴리즈 노트 포맷)
